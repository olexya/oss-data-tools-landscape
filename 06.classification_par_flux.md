---

## Introduction

Les architectures de traitement de données modernes peuvent être classées selon trois paradigmes principaux basés sur la **latence** et la **fréquence** de traitement des données:

### 1. **Streaming (Temps Réel)**
Traitement continu des données à la volée, événement par événement, avec une latence de l'ordre de la **milliseconde à la seconde**.

**Caractéristiques:**
- Latence ultra-faible (< 1 seconde)
- Traitement événementiel continu
- Idéal pour alertes temps réel, détection de fraude, IoT
- Architecture event-driven
- Exemples d'usage: Monitoring en temps réel, trading algorithmique, détection d'anomalies

### 2. **Micro-Batching (Quasi Temps Réel)**
Traitement par petits lots de données à intervalles très courts, avec une latence de l'ordre de **quelques secondes à quelques minutes**.

**Caractéristiques:**
- Latence faible (1 seconde - 5 minutes)
- Accumulation de données sur de courts intervalles
- Compromis entre throughput et latence
- Idéal pour analytics quasi temps réel
- Exemples d'usage: Tableaux de bord temps réel, agrégations récentes, métriques business

### 3. **Batching (Traitement par Lots)**
Traitement de grands volumes de données à intervalles planifiés (horaire, quotidien, hebdomadaire), avec une latence de l'ordre des **minutes à heures**.

**Caractéristiques:**
- Latence élevée acceptable (> 5 minutes)
- Traitement de volumes importants
- Optimisé pour le throughput
- Idéal pour analyses historiques, ETL, rapports
- Exemples d'usage: Rapports quotidiens, data warehousing, ML training

---

## Matrice de Classification

### Critères de Classification

| Critère | Streaming | Micro-Batching | Batching |
|---------|-----------|----------------|----------|
| **Latence** | < 1 seconde | 1 sec - 5 min | > 5 minutes |
| **Taille des lots** | 1 événement | Petits lots (KB-MB) | Grands lots (GB-TB) |
| **Fréquence** | Continue | Haute (secondes) | Basse (heures/jours) |
| **Use Case** | Temps réel | Quasi temps réel | Analytique historique |
| **Complexité** | Élevée | Moyenne | Faible |
| **Throughput** | Moyen | Moyen-Élevé | Très élevé |

## Architectures Recommandées

### Architecture Lambda (Batch + Streaming)

```
Ingestion:        Kafka + Airbyte
Stream Layer:     Flink → Druid/Pinot
Batch Layer:      Spark → Iceberg/Delta Lake
Query:            Trino (data lake) + Druid (real-time)
Viz:              Superset + Grafana
```

**Use Cases:** Analytics temps réel + historique, e-commerce, IoT

---

### Architecture Kappa (Streaming Only)

```
Ingestion:        Kafka
Processing:       Flink → Materialize
Storage:          Hudi/Paimon
Query:            ClickHouse
Viz:              Grafana + Dash
```

**Use Cases:** IoT temps réel, monitoring, détection fraude

---

### Architecture Batch Moderne

```
Ingestion:        Airbyte
Orchestration:    Airflow/Dagster
Processing:       dbt + Spark
Storage:          Iceberg on MinIO
Query:            Trino
Viz:              Superset
```

**Use Cases:** Data warehouse moderne, BI, ML

---

### Architecture Micro-Batching

```
Ingestion:        NiFi (1-minute batches)
Processing:       Spark Structured Streaming
Storage:          Delta Lake
Query:            DuckDB + StarRocks
Viz:              Metabase (auto-refresh)
```

**Use Cases:** Dashboards quasi temps réel, KPIs business

## Guide de Sélection

### Choisir selon la Latence Requise

| Besoin | Latence Cible | Mode Recommandé | Stack Exemple |
|--------|---------------|-----------------|---------------|
| Alertes critiques | < 100ms | Streaming | NATS + Flink + Druid |
| Dashboards temps réel | < 1 sec | Streaming | Kafka + Flink + ClickHouse |
| Analytics quasi temps réel | 1-5 min | Micro-Batching | Spark Streaming + Delta Lake |
| Rapports horaires | 1 hour | Batching | Airflow + dbt + Iceberg |
| Rapports quotidiens | 24 hours | Batching | Airbyte + Spark + Parquet |

---

### Choisir selon le Volume

| Volume Quotidien | Mode Recommandé | Justification |
|------------------|-----------------|---------------|
| < 10 GB | Streaming/Micro-Batching | Peu de données, privilégier latence |
| 10-100 GB | Micro-Batching | Bon compromis latence/coût |
| 100 GB - 1 TB | Micro-Batching/Batching | Optimiser throughput |
| > 1 TB | Batching | Coût et efficacité primordiaux |

---

### Choisir selon le Use Case

| Use Case | Mode | Stack Recommandé |
|----------|------|------------------|
| **Monitoring système** | Streaming | Vector → ClickHouse → Grafana |
| **Détection fraude** | Streaming | Kafka → Flink → Druid → Alertes |
| **E-commerce analytics** | Lambda | Kafka + Flink (stream) + Spark (batch) |
| **Data Warehouse** | Batching | Airbyte → dbt → Iceberg → Trino |
| **IoT sensors** | Streaming | MQTT → Kafka → Flink → TimescaleDB |
| **BI Dashboards** | Micro-Batching | dbt → StarRocks → Superset (refresh 5min) |
| **ML Training** | Batching | Spark → Parquet → Notebooks |
| **Customer 360** | Lambda | CDC (stream) + ETL (batch) → Delta Lake |
